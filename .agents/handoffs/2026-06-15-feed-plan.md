# 工作台 Feed ④ —— 实施方案（决策已锁定，待落地）

状态：**设计定稿、决策锁定，尚未实现**。下次会话据此直接进 TDD。
更新时间：2026-06-15
依赖：③ 政策匹配已合并 main（PR #13）；本方案复用 `PolicyMatchService` 作计算引擎。

## 目标

主线第④步：把 ③ 匹配出的可申报政策以**持久化信息流**呈现在企业工作台，主动标出「新增」，无需用户每次主动点算。③ 解决"算得出"，④ 解决"系统替你盯、有新的顶上来"。

## 决策（2026-06-15 与用户敲定）

1. **物化**：新建 `policy_matches` 表落地 Feed（③ 是即时不落表，④ 必须落表才能判新增/未读计数/免重算）。
2. **触发 = (a)+(b) 组合，不引调度库**：
   - (a) 抓取政策入库后 → 重算**当前租户**（不是所有租户，省成本）。
   - (b) 企业档案保存后 → 重算该租户。
   - (c) Feed 页「重新匹配」按钮手动重算（兜住跨租户：别人抓的新政策，本租户点一下就拉到）。
3. **状态机**：`unread | read | applied | ignored`（未读/已读 + 已申报/忽略）。
4. **推送形态**：仅站内 —— 工作台 Feed 页 + 左栏入口未读红点/计数。外部推送（邮件/IM）后置，不阻塞。
5. **`/matches` 由 `/feed` 取代**：③ 的即时预览页 `/matches` 删除；`/policies/match` 端点**保留**作计算引擎被 FeedService 复用。避免两个几乎一样的页面。
6. **入口命名**：左栏叫「**工作台**」。

### 已知副作用（已用默认兜住）
公开政策库全局共享，按"抓取后只重算当前租户"，**其他租户 Feed 不会自动刷新**，需其自己改档案或点「重新匹配」。Feed 页保留手动按钮即可填补，省成本不漏。

## 实施计划

### 数据模型 `policy_matches`（基础设施层 ORM + 迁移）
```
id            uuid pk
tenant_id     str (索引)
policy_id     str
-- 计算快照(展示用,Feed 列表单表查询免 N+1,政策变动也稳定) --
title / issuer / publish_date / source_url / region
score / structured_score / semantic_score
matched_terms jsonb
reasons       jsonb
-- 状态机 --
status        enum(unread|read|applied|ignored)  默认 unread
created_at / updated_at
UNIQUE(tenant_id, policy_id)   -- 一租户一政策一条,重算 upsert
```
迁移 down=`e6f7a8b9c0d1`，纯新增向后兼容。

### 重算语义（关键：新增判定 + 幂等 + 不冲掉用户状态）
`FeedService.recompute_for_tenant(tenant_id)`：跑 `PolicyMatchService.match_for_tenant` →
- 已存在 `(tenant, policy)`：更新分数/理由快照，**保留 status 与 created_at**（用户标过的不被重置）。
- 不存在：插入 `unread` → 即"新增"，驱动红点。
- 跌出候选的旧条目：**保留不删**（可能已申报/想留存）。
- 未读数 = `count(status=unread)`；"自上次以来新增" 天然等价 unread。

### 应用服务 `FeedService`
- `recompute_for_tenant(tenant_id)` → {new, updated}
- `list_feed(tenant_id, status, page, page_size)`（分页，单表）
- `unread_count(tenant_id)`
- `set_status(tenant_id, match_id, status)`（read/applied/ignored，校验租户归属）
- `mark_all_read(tenant_id)`（打开页面清红点）

### 触发接线
- (a) `POST /policies/ingest` 后台任务链：`ingest` 完成后追加 `recompute_for_tenant(current_user.tenant_id)`（需把 tenant_id 传进后台任务）。
- (b) `PUT /enterprise-profile` 端点返回后排后台 `recompute_for_tenant`（**在 endpoint 触发，服务层不互相耦合**）。

### 端点 `feed_routes.py`（前缀 `/feed`，所有登录用户，限当前租户）
- `GET /feed?status=&page=&page_size=`
- `GET /feed/unread-count`
- `POST /feed/recompute`（手动，当前租户）
- `POST /feed/{match_id}/status`（body: read|applied|ignored）
- `POST /feed/mark-read`（打开页面批量 unread→read）

## 风险 / 注意
- 重算 upsert 必须保留用户 status，否则已申报/忽略会被冲掉（重算语义已规定）。
- 后台任务在 API 进程内跑（沿用 ② ingest 模式），租户多/政策多时重算耗时；MVP 单租户重算可接受。
- 未读红点需前端在导航挂载 + 重算后刷新 `unread-count`（无 WebSocket，先按需拉取/轮询）。
