# 资质申报机会 ⑥ —— 决策固化 + 资质种子目录

状态：**Phase 1 已实现（目录+匹配能力①+接入 ④ Feed），PR #21 待确认合并**；能力②差距分析、③材料指引后续单独迭代。
更新时间：2026-06-16
依赖：① 企业档案(已合,含 `qualifications`/`tech_domains`/`keywords`/`industry`/`scale`/`region` 字段)；③ 政策匹配引擎(已合,`policy_matcher.py`)；④ 工作台 Feed(方案已锁,实现中)。

## 背景

企业方反馈:除政策外,同样关注**资质相关申请**和**相关比赛**。这正是 ① 决策里主动推迟的"赛事后置"部分。本次把"机会(opportunity)"从单一"政策文件"扩成多类:政策 / 资质 / (后置)比赛。

## 决策（2026-06-15 与用户敲定）

1. **排期**:先做完 ④ 工作台 Feed,再扩机会类型。资质是 Feed 之后的下一块、本期重点;比赛暂缓。
2. **资质 = ④ Feed 的第二类内容源**(opportunity 抽象)。Feed 先按政策跑通,资质接入只多一个 `type` 维度,匹配→Feed→未读红点整套复用。**④ Feed 落地时数据模型预留 `type`/来源标识扩展位,避免回头改表。**
3. **数据源 = 「资质目录」打底,不爬。** 与比赛最大区别:资质有限、相对稳定、且本身由政策文件定义(高企认定/专精特新等各有"认定办法")。维护一份结构化资质目录(几十条),零爬取风险、质量可控——吸取 ①b 因无可靠源翻车的教训。
4. **资质目录首版来源 = 方案 A**:由我(AI)按 新吴区/江苏/国家级 整理种子目录,企业方校对补充。本文「资质种子目录」即首版。
5. **三个能力**(企业方要求,层层递进,均复用现有积木):
   | 能力 | 实现 | 复用 |
   |---|---|---|
   | ① 匹配可申报资质 | 企业档案 × 资质目录条件,算"够条件/接近够" | ③ `policy_matcher` 命中/打分逻辑 |
   | ② 条件解读 / 差距分析 | Agent 拿"资质条件 + 企业档案 + 政策原文"产出差距清单 | 现有 Agent + KnowledgeBaseTool(政策已在公开库) |
   | ③ 材料/流程指引 | 目录带材料清单/流程/时间节点;Agent 进一步组装 | 衔接 ⑤ 报告链路 |
6. **比赛暂缓**:现状主要靠**公众号**推送(微信生态封闭、基本爬不动,硬抓=①b 第二季)。可行路径是**可配置的赛事官网源列表**(创新创业大赛、工信/科技口、行业协会官网),逐源验证"接口可逆向、可结构化"再投。挂起,资质做完再开调研小分支。

## 资质种子目录（首版 · 待企业方/官方办法校对）

> **重要校对说明**:下列「核心条件」「材料」「时间」均为**结构性概要**,具体数值/比例/名额/申报窗口**以当年官方最新办法为准**(资质条件逐年微调)。`政策依据` 给出认定办法名称/文号,落地时关联到公开政策库对应原文(或补抓)。本目录用于驱动「匹配 + 差距分析 + 指引」,数值类条件须在实现前逐条核校。
>
> 字段对齐未来 DB schema(opportunity/qualification):`name / level / issuer / category / key_conditions / materials / timing / policy_basis / benefit / match_signals`。
> `match_signals` = 与企业档案匹配用的信号(行业/技术域/规模门槛/已持资质前置等)。

## 实施计划（待 ④ Feed 完成后启动）

### 数据模型
- 复用 ④ Feed 的 opportunity 抽象;新增 `qualifications`(或 `opportunity` 加 `type=qualification`)结构化目录表,字段见上(name/level/issuer/category/key_conditions/materials/timing/policy_basis/benefit/match_signals)。
- 种子数据:把本目录转为 seed(JSON/迁移内插入或单独 seed 脚本),首版人工维护、后续可半自动。

### 能力①匹配
- 复用 `policy_matcher`:企业档案 `qualifications/tech_domains/keywords/industry/scale/region` × 资质 `match_signals/key_conditions` 命中打分;输出"可申报 / 接近可申报(差 N 项)"。规模门槛、注册地、前置资质(如小巨人需省专精特新)作硬/软信号。
- 结果进 ④ Feed,`type=qualification`,与政策同流呈现。

### 能力②差距分析
- 新增 Agent 工具或服务:输入(资质 key_conditions + 企业档案 + 关联政策原文 via KnowledgeBaseTool)→ 输出逐条"满足/不满足/缺材料"差距清单。复用聊天 Agent 链路。

### 能力③材料/流程指引
- 目录自带 materials/timing;Agent 可据档案组装个性化清单与时间表,衔接 ⑤ 报告。

- **UI**：`/qualifications` 资质机会页（列表+详情弹窗）、`components/qualification-detail.tsx`（免责声明 banner 复用）、Feed 类型徽章 + 资质详情分流、左栏「资质机会」入口。
- **测试**：matcher 8 + service 4 + catalog 契约 6 + Feed 接入 2。

## 风险 / 注意

- **数值类申报条件会逐年变**:目录里的比例/门槛/窗口期为概要,**严禁当权威输出**,实现前须校对并标注"以官方最新办法为准 + 末次核对日期"。
- 资质多由政策定义,`policy_basis` 应尽量关联到公开政策库原文(无则补抓),供差距分析取证。
- 比赛走公众号、微信封闭,**不要尝试硬爬微信**;只对可逆向的官网源投入。
- 目录维护是长期人工成本,首版控制在高价值高频(本文约 24 条)即可,勿一次铺满。
