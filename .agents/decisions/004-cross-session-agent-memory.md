# ADR 004：跨会话 Agent 记忆（长期记忆第 2 层）

状态：已决定（2026-06-20）
关联：[[002-rag-as-agent-tool]]、handoff `2026-06-20-agent-enterprise-context`

## 背景

#59 给 Agent 装上了"实体记忆"——把企业档案渲染成 `<enterprise_profile>` 注入系统提示词，
Agent 因此知道"我服务于哪家企业"。但那只是**结构化档案**这一层。Agent 仍然：

- 只有**会话内消息历史**（`sessions.memories` JSONB，planner/react 各一份）这层短期上下文窗口；
- **没有任何跨会话记忆**：用户上次会话里说过的事实/偏好（"我们今年新增了 3 项发明专利"、
  "回答请简洁、别用列表"、"我们重点想申报专精特新"），换个会话就忘得一干二净。

这是"带工具的 chatbot"与"真 Agent"的分水岭。#59 的 handoff 已把这层定性为 **ADR 级独立特性**。

## 决策

**引入按租户的长期记忆条目库；写入走 Agent 主动调用的 `remember` 工具；召回走会话启动注入。**

完全复刻 #59 实体记忆的"渲染纯函数 + 会话启动注入"范式，只是数据从"档案表"换成"记忆条目表"，
并新增一条 Agent 自主写入的路径。

### 两半结构

| 维度 | 决策 | 理由 |
|---|---|---|
| 召回(读) | 会话启动时按 `tenant_id` 取记忆条目 → `render_memory_context` 渲染成 `<long_term_memory>` 块 → 注入 planner/react 首条 system 消息 | 复刻 `enterprise_context` 注入链，零风险、与 #59 同构 |
| 写入 | **工具式**：`MemoryTool.memory_save(content)`，Agent 判断"用户陈述了值得长期记住的事实/偏好"时主动调用 | 最 agentic、可在 UI 显式呈现"已记住"；可控、低风险，不引入隐式后处理流水线 |
| 存储 | 新表 `agent_memories`，每租户多行事实条目 | 简单；条目有界 |
| 检索 | **不用向量**，按时间倒序取最近 N 条全量注入（N=`MEMORY_RECALL_LIMIT`，默认 30） | KISS；避开"全库语义检索"复杂度与 embedding 依赖。条目规模真大时再上向量召回 |
| 范围 | **按租户**（与实体记忆/企业为主体一致） | 同一企业的任意会话都能召回；`created_by` 仅作溯源 |

### 为什么写入用工具而非自动抽取

自动后处理（会话结束跑 LLM 抽事实）召回覆盖更全，但：① 隐式、不可见、难 demo；
② 多一条后台 LLM 调用与失败面；③ 抽取质量难控、易记噪音。工具式让**模型显式决定**记什么，
SSE 上还能渲染"🧠 已记住：…"卡片。自动抽取留作后续增强（可作为 `memory_save` 的补充来源）。

## 影响

- 新表 `agent_memories`（纯新增，向后兼容），迁移 head 接 `b9c0d1e2f3a4`。
- `planner_react` 会话启动多读一次记忆并注入；`base.py` 系统上下文拼接从"仅企业档案"
  泛化为"企业档案 + 长期记忆"两段。
- 新增 `MemoryTool` 进工具列表；`system.py` 加 `<memory>` 指引（何时记、记忆块为权威背景）。
- 新增 `GET /agent-memories`、`DELETE /agent-memories/{id}`（限当前租户）+ 前端「Agent 记忆」页查看/删除。

## 已知边界

- 与 #59 一致：系统提示词只在**会话首条消息**落库一次，故记忆注入**只对新会话生效**，
  存量旧会话不回溯（新建会话即可用上新记忆）。
- 记忆为租户共享：同企业多成员共用一份长期记忆（符合"企业为主体"，但偏个人化的偏好会共享）。
- 不做语义召回与自动过期；条目无界增长时靠管理页人工清理 + 注入侧 N 上限兜底。

## 未完成 / 后续

- 自动抽取（会话结束 best-effort 抽事实补 `memory_save`）。
- 语义召回（条目向量化，按当前话题召回最相关 N 条）。
- 记忆冲突/更新策略（"专利从 3 项变 5 项"时的覆盖而非追加）。
